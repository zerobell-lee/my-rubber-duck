# AI Rubberduck — 설계 문서

**날짜**: 2026-04-24
**상태**: 초안 (사용자 리뷰 대기)

## 1. 개요

Claude Code 슬래시 커맨드 플러그인. 사용자가 특정 주제(주로 자기 프로젝트의 일부분)를 **신입 직원 역할의 AI에게 설명**하게 하여, 설명 근육을 기르고 이해도의 빈틈을 드러낸다. 세션 종료 시 AI가 레포를 팩트체크해 **갭 중심 피드백**을 제공한다.

Rubberduck 기법의 원리 — "남에게 설명하면서 스스로 깨닫는다" — 를 AI 시대에 맞춰 재구성한 것. AI에게 "답을 받는" 게 아니라, AI를 **경청하는 초심자의 자리**에 놓고 사용자가 설명의 주체가 되게 한다.

## 2. 개발 동기

- 프로젝트 규모가 커지면서 한 사람이 전 도메인을 숙지하기 어려워짐
- AI가 개발의 대부분을 맡으면서 인간 개발자의 도메인 이해도가 점진적으로 낮아짐
- 신규 입사자 온보딩이 사라지면서 현업자가 "설명을 통해 다시 배우는" 기회를 잃음
- 이 도구는 **AI를 신규 입사자의 자리에 놓아** 그 기회를 복원한다

## 3. 목표

- 사용자가 프로젝트 특정 영역에 대해 자신의 이해도를 **객관적으로 확인**
- "안다고 착각하던" 부분을 드러내기 (레포 팩트체크)
- 설명·전달 능력 훈련

## 4. 비범위 (Non-goals)

- **버그 해결·디버깅 보조**: AI가 직접 해결하는 게 더 빠름. Rubberduck의 원래 목적과 다름.
- 실시간 음성 대화
- 외부 LLM 제공사 연동 (Claude Code 내부에서 완결)
- 학습 성과의 팀·조직 단위 집계 (개인 도구로 시작)

## 5. 핵심 결정사항

### 5.1 유즈케이스

**학습/이해도 점검만 (one use case).**

초기에 "문제 해결 rubberduck"도 고려했으나 폐기. AI 시대에 기술적 막힘은 AI가 직접 해결하는 게 낫고, "판단의 막힘"은 rubberduck이 아닌 Socratic coach에 더 가까움. 학습 유즈케이스 안에 rubberduck 기법의 본질(설명을 통한 자기 명료화)이 이미 포함됨.

### 5.2 AI의 레포 접근 모델: B (Grading-time reveal)

- **Phase 1–2 (Intake, Q&A)**: AI는 레포에 접근하지 않음. 일반 기술 지식만 사용.
- **Phase 3 (Grading)**: AI가 레포를 읽고 사용자 설명과 대조하여 팩트체크.

**근거**: Q&A 중엔 role-play의 교육적 가치(설명 근육 훈련)를 유지해야 함. 채점 시점엔 "내가 잘못 알고 있던 것"을 정확히 짚어주는 실용적 가치가 role-play 일관성보다 크다.

**백로그로 넘긴 대안**:
- **C (Silent pre-read)**: AI가 시작 전 레포 스캔. 질문은 날카로워지나 확증 편향·유도심문 위험.
- **B+ (표면 스캔)**: 파일명·디렉토리 구조만 사전 스캔, 구현은 안 봄. C와 B의 절충.

### 5.3 채점 형식: C (갭 중심 피드백)

점수 없음. "잘 설명함 / 흔들림 / 사실 오류 / 다음에 볼 것" 구조. 맨 위에 한 줄 총평만.

**근거**: 단일 점수는 학습 도구로서 저가치 (다음에 뭘 할지 모호). 루브릭은 주제마다 적절한 축이 다름. 갭 중심은 **행동으로 이어지는** 피드백이고, B 모델의 팩트체크와 궁합이 가장 좋음.

### 5.4 세션 종료: D (AI 제안 + 사용자 결정)

AI가 충분히 파악했다고 판단하면 "이 정도면 충분한 것 같은데, 더 짚고 싶으신 부분 있으세요?"로 제안. 사용자가 `done`/`채점`/`끝` 등으로 종료.

**근거**: 고정 질문 수는 주제 크기에 안 맞고, AI 판단만 맡기면 과도하게 끈질겨질 수 있고, 사용자 결정만 맡기면 편향으로 일찍 끝남.

### 5.5 형태: 슬래시 커맨드 (단독)

`/rubberduck [topic] [depth]`. 스킬 자동 활성화 아님.

**근거**: Rubberduck 세션은 항상 사용자가 의도적으로 시작하는 모드. 맥락 기반 자동 활성화 부적합.

### 5.6 배포: 오픈소스 플러그인

`.claude-plugin/plugin.json` 매니페스트 포함, GitHub 공개. 다른 팀·개인이 설치해 사용 가능.

### 5.7 언어: 영어 메타데이터/프롬프트, 대화는 자동 매칭

- README, `plugin.json`, 커맨드 프롬프트는 **영어**
- 사용자와의 실제 Q&A는 Claude가 사용자 언어를 자연스럽게 매칭

**근거**: LLM의 instruction-following은 영어에서 가장 안정적. 오픈소스 저변 확대에도 유리. 한국어 사용자도 UX는 한국어로 유지됨.

### 5.8 영속성: 채점 결과만 저장 (기본 켬, opt-out)

- 저장 경로: `<user-project>/.rubberduck/sessions/YYYY-MM-DD-[slug].md`
- 저장 내용: Phase 3 채점 결과 마크다운만. Q&A 전체 대화는 저장 안 함.
- `--no-save` 플래그로 끄기 가능
- `.rubberduck/`가 `.gitignore`에 없으면 경고

**근거**: 학습 도구로서 복기 자산은 필요. Q&A 전문 저장은 민감한 아키텍처 내용이 평문으로 남는 문제 — 코어 피드백만 보존.

### 5.9 세션 시작 UX: C (인자 생략 시 대화형 프롬프트)

- `/rubberduck` → AI가 주제, 그 다음 depth를 순차로 질문
- `/rubberduck auth system` → depth만 질문
- `/rubberduck auth system deep` → 바로 Phase 2 진입

**Depth 단계**:
- `shallow`: 3–5 질문
- `medium` (기본값): 6–10 질문
- `deep`: 10+ 질문, 엣지 케이스·설계 근거 포함

## 6. 아키텍처

**선택: Monolithic 커맨드 프롬프트.**

단일 `commands/rubberduck.md` 파일이 4개 phase의 지침을 모두 포함. 메인 Claude 컨텍스트에서 실행. 훅·상태 파일·서브에이전트 없음.

**탈락 대안**:
- **서브에이전트 분리**: 서브에이전트는 사용자와 실시간 turn-taking에 부적합. Q&A가 본질적으로 왕복 대화라 패턴 불일치.
- **훅 + 상태 파일**: "Phase 2에서 레포 접근 차단"을 기계적으로 강제할 수 있으나, 오픈소스 플러그인에 훅 설치 요구는 UX 부담. 프롬프트 규율로 충분.

## 8. 세션 Phase 상세

### Phase 1: Intake

**목적**: 주제와 depth 확정, 사용자가 자연스럽게 설명을 시작하게 유도.

**동작**:
1. 인자에서 주제·depth 파싱
2. 누락된 항목 순차적으로 질문
3. Depth 확정 후 "편하신 부분부터 시작해주세요" 같은 open-ended 안내

**제약**:
- ❌ `Read` / `Grep` / `Glob` / `Bash` 사용 금지

### Phase 2: Q&A (신입 role-play)

**목적**: 사용자가 설명을 통해 자신의 이해도를 스스로 검증하고 빈틈을 드러내게 함.

**AI 페르소나**: "이 팀에 오늘 합류한 신입 개발자". 이 레포를 처음 봄. 일반 기술 지식(분산 시스템, DB, 인증, 보안, 성능 등)은 있음.

**질문 전략**:
- 사용자 설명에서 논리적으로 빈틈이 보이는 지점 파고들기
- 일반 도메인 원칙 기반 날카로운 질문 (예: "고가용성은 어떻게 확보하나요?", "이 쿼리 패턴에서 N+1은 안 생기나요?", "토큰 폐기 시나리오는?")
- 사용자가 막히면 힌트성 질문으로 한 단계 내려가기 ("제 짐작으론 X일 것 같은데 맞나요?")
- 사용자가 2번 이상 "모르겠다"고 답하면 그 토픽을 갭으로 기록하고 넘어감

**종료 트리거**:
- AI 판단: 주제의 주요 하위 영역이 충분히 다뤄짐 → "이 정도면 파악된 것 같은데, 더 짚고 싶은 부분 있으세요?"
- 사용자 신호: `done` / `채점` / `끝` / `그만` / "채점해줘" / "이제 됐어" 등 → 즉시 Phase 3
- Depth별 권장 질문 수 초과 시 AI가 종료 제안 빈도 높임

**제약**:
- ❌ `Read` / `Grep` / `Glob` / `Bash` 사용 금지
- ❌ 알고 있는 것처럼 유도심문하지 말 것 ("혹시 Redis 쓰시나요?" ❌ → "캐시 계층은 어떻게 구성하셨어요?" ✅)
- ❌ 레포 지식을 드러내지 말 것

### Phase 3: Grading (팩트체크)

**목적**: 사용자 설명을 실제 코드와 대조하여 갭 식별.

**동작**:
1. 레포 접근 허용 (`Read`, `Grep`, `Glob`)
2. Phase 2에서 사용자가 언급한 주요 주장들을 코드로 확인
3. 누락한 중요 토픽(도메인 일반 원칙) 식별
4. 아래 형식으로 출력

**출력 형식**:
```markdown
# [주제] — 이해도 피드백 (YYYY-MM-DD)

**한 줄 총평**: [강점 요약 + 가장 큰 갭]

## ✅ 잘 설명하신 부분
- [구체 항목]

## ⚠️ 흔들리신 부분
- [구체 항목 — 설명이 모호했거나 일부 누락된 주제]

## ❌ 사실 확인 필요
- "[사용자 발언 요약]" — 실제 `path/to/file.ts:LN`에선 [실제 동작]입니다.

## 💡 다음에 살펴볼 만한 것
- [제안 토픽 2-3개]
```

### Phase 4: Save

**동작**:
1. `--no-save` 플래그가 없으면 Phase 3 출력을 `.rubberduck/sessions/YYYY-MM-DD-[slug].md`에 저장
2. 파일명의 `[slug]`는 주제에서 생성 (공백·특수문자 → 하이픈, 소문자)
3. `.rubberduck/`가 `.gitignore`에 없으면 경고 메시지 출력 및 추가 제안

## 9. 핵심 동작 규칙 (프롬프트 명시 항목)

1. **Role-play 일관성**: Phase 2 내내 AI는 "이 레포를 처음 보는 신입". 알고 있는 정보도 사용자 입을 통해 들은 것처럼 처리.
2. **Phase별 tool 제약**: Phase 1–2 동안 `Read`/`Grep`/`Glob`/`Bash` 호출 금지. Phase 3 진입 시점에만 해제.
3. **질문 품질**: 일반 질문 ≠ 뻔한 질문. "트랜잭션 경계는?", "장애 시 복구 절차는?", "동시성 제어는?" 같은 도메인 원칙 기반의 날카로움 유지.
4. **사용자 리듬 존중**: 사용자가 "모르겠다"고 해도 바로 답 주지 말고 맥락 질문으로. 2회 이상 모른다고 하면 갭으로 기록 후 넘어감.
5. **언어 자동 매칭**: 사용자 언어를 자연스럽게 따름.
6. **채점의 친절함**: 갭을 지적하되 비난조 아님. "모르셨군요"가 아니라 "이 부분 보시면 좋을 것 같아요".

## 11. 향후 백로그

- **옵션 C (Silent pre-read) 모드 추가**: 사용자가 "질문이 너무 일반적이다"고 피드백하면 추가. 편향 리스크 인지.
- **옵션 B+ (표면 스캔)**: C의 완화 버전.
- **진도 추적**: 여러 세션 피드백 집계 → "반복해서 흔들리는 영역" 리포트.
- **주제 제안**: 레포 구조를 보고 "이런 주제로 학습 세션 열어볼 수 있어요" 제안.
- **팀 공유 모드**: 익명화된 세션 요약을 팀 지식 갭 대시보드로 (조직 배포 시).
